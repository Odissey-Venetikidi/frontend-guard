# Гайд по фронтенду для LLM-агента: ЯДРО + ПРОФИЛЬ ПРОЕКТА

Этот документ — онбординг для LLM-агента, который пишет фронтенд. Он состоит из двух частей:

- **ПРОФИЛЬ ПРОЕКТА** (раздел 0) — короткий заполняемый блок. Настраивается ОДИН раз под конкретный проект. Все имена, пути, префиксы и инвентарь примитивов — здесь.
- **ЯДРО** (разделы 1-11) — framework-neutral правила, дисциплина, философия. Они написаны через РОЛИ и ПРИНЦИПЫ; конкретные имена сущностей всегда берутся из Профиля.

**Чтобы перенести этот гайд в новый проект, ты переписываешь ТОЛЬКО раздел 0. Ядро не трогаешь.** Это и есть вся модель ЯДРО+ПРОФИЛЬ: ядро описывает «как думать», Профиль — «как это называется здесь». Если в ядре написано «примитив-кнопка», «UI-слой», «токен spacing» — конкретное имя подставляется из Профиля.

**Принципы языко-независимы.** «Фронтенд» здесь = любой UI-слой: web (React/Vue/Svelte), mobile (Flutter/Dart, SwiftUI/Swift), desktop. Композиция из примитивов, нет стилей/хардкода вне UI-слоя, единый типизированный клиент для I/O, СТОП-и-спроси — справедливы для любого языка. Профиль фиксирует язык и его имена (виджет вместо компонента, ThemeData/asset-catalog вместо CSS-токенов, типизированный repository/networking вместо api-клиента); ядро не меняется. Автоматический линтер `guard.mjs` покрывает web/Flutter/SwiftUI; для остальных языков принципы держатся гайдом, аудитом и ревью (плюс `customRules` в конфиге).

## 0. ПРОФИЛЬ ПРОЕКТА (заполнить под проект ОДИН раз)

> Заполни каждый плейсхолдер `<...>` под свой проект. В комментариях даны два примера заполнения: **React/Next** и **Vue** — чтобы было видно, как адаптировать. После заполнения удали неподходящий пример (оставь только свой стек). Если в проекте чего-то нет (например, нет карт) — оставь прочерк, но **не удаляй строку**, чтобы агент видел, что вопрос был задан. Незаполненная роль примитива = «примитива нет» = триггер СТОП (раздел 2) при первой надобности.

### 0.3 Префиксы дизайн-токенов (рекомендуемая конвенция; держи ОДНУ конвенцию на проект)

> Имена utility-классов и токенов НЕ хардкодятся в ядре — ядро ссылается на эти плейсхолдеры (`<type-class>`, `<text-color-class>`, `<tabular-class>`, `<color-token>` и т.д.). Если в проекте другие префиксы — переопредели здесь, ядро останется верным.


# ЯДРО (framework-neutral; имена берём из Профиля)

## 1. ГЛАВНЫЙ ПРИНЦИП (сердце документа)

**Фронтенд = композиция из готовых примитивов, а не свободное написание HTML/CSS.**

Почти всё, что хочется написать руками — кнопка, селект, модалка, таблица, заголовок, отступ, цвет, размер шрифта — уже существует как примитив (см. инвентарь, Профиль 0.7) или токен (Профиль 0.3). Задача агента — **найти и использовать существующее, а не создавать параллельное.**

Нет нужного примитива / токена / варианта — **СТОП**. Не писать кастомный CSS как обходной путь. Остановиться и сказать пользователю: «нужен паттерн X, в UI-слое его нет», предложить расширить примитив или завести новый — и **дождаться решения**.

Это главное правило. Его нарушение ломает визуальный язык всего продукта.

## 2. ЖЁСТКИЕ UI-ПРАВИЛА (non-negotiable)

### 2.1 СТОП-триггеры (выучить наизусть)

> Любой из этих сигналов = немедленная остановка. НЕ обходить, НЕ «прикрутить временно». Сказать пользователю по-русски: «нужен паттерн X, в UI-слое его нет», предложить расширение/токен/иконку/вариант — и **дождаться решения**. Правило выше соблазна «быстро прикрутить».

| СТОП-триггер (ситуация) | Что НЕ делать | Что делать |
|---|---|---|
| Нет нужного примитива под роль | Писать кастомный HTML/CSS как обходной путь | СТОП → предложить расширить примитив или завести новый → ждать решения |
| Нет нужного токена (цвет/spacing/radius/шрифт/тень) | Вписать hex/px прямо в компонент | СТОП → завести токен (в `:root` + override темы) → ждать |
| Нет нужной иконки | Сырой юникод-кодпоинт или `fa-*` класс | СТОП → предложить добавить с kebab-case именем и вариантом (light/solid/brand) → ждать |
| Нет нужного варианта (color/variant/size) | Изобрести ad-hoc проп или инлайн-стиль | СТОП → предложить расширить API примитива по конвенциям §4 → ждать |
| Паттерн не выразим существующими примитивами | Свободный CSS в странице/фиче-блоке | СТОП → предложить новый примитив → ждать |

### 2.2 Только канонические примитивы из UI-слоя

НИКОГДА не использовать сырые нативные интерактивные элементы: нативную кнопку, нативный select, нативные `input[type=checkbox|tel|date]`, нативный диалог. Вместо них — соответствующий примитив из инвентаря (Профиль 0.7).

**Таблица: роль → примитив (имя из Профиля) → НИКОГДА**

| Хочешь | Бери роль (Профиль) | НИКОГДА / анти-паттерн |
|---|---|---|
| Кнопка с текстом | примитив-кнопка `<btn>` | нативный `button`, `div` c onClick |
| Кнопка без URL, визуально-ссылка (напр. «Скачать») | `<btn>` с `variant=link` (это ACTION, не навигация) | `<a>` без href |
| Только иконка-действие | icon-кнопка `<icon-btn>` (+ `aria-label`) | `button class="icon-btn"`, кликабельная голая иконка |
| Выбор одного из списка | select `<select>` (`:options` + биндинг значения) | кастомный trigger+popup, нативный `select` |
| Выбор нескольких | multi-select `<multiselect>` | набор чекбоксов вручную |
| Текстовый ввод | поле-обёртка `<field>` | голый `input` без обёртки |
| Список с пагинацией/поиском/сортировкой | таблица-список `<table>` | своя таблица руками |
| Inline-ссылка навигации (внешний URL, mailto, tel, якорь) | ссылка `<link>` (external) | свой `a` со стилями |
| Внутренняя SPA-навигация | навигационный примитив фреймворка `<nav-link>` | ручной роутинг через `a href` |

**Про оси кнопки** (имена пропов — из Профиля): `color` — семантика (default/primary/success/warning/danger), `variant` — визуальная форма (solid/light/link). `variant=link` — это affordance «выглядит как ссылка», но это **ДЕЙСТВИЕ без URL** (напр. «Скачать»), не навигация. Реальная навигация — всегда роль ссылки `<link>` или навигационный примитив `<nav-link>`.

### 2.3 Диалоги только через confirm/alert-хелпер (Профиль 0.5)

Подтверждения и сообщения — через `<confirm-fn>` / `<alert-fn>` / `<alert-error-fn>` из `<confirm-module>`. **НИКОГДА** не использовать `window.confirm` / `window.alert` — они обходят платформенный диалог и выглядят чужеродно.

| Нужно | Вызов (имя из Профиля) | Заметка |
|---|---|---|
| Подтверждение (да/нет) | `await <confirm-fn>({title, text?, confirmText?, cancelText?, danger?}) -> Promise<boolean>` | деструктив — `danger: true` |
| Сообщение (ок) | `await <alert-fn>({title, text?}) -> Promise<void>` | — |
| Ошибка с трассировкой | `await <alert-error-fn>('Не удалось…', err)` в `catch` | вытащит message + request-id (Профиль 0.4) + серверный код из типа ошибки |

Прямая модалка `<modal>` остаётся для форм / picker / multi-step — всего, что **НЕ** сводится к «заголовок + текст + до двух кнопок».

### 2.4 Иконки (Профиль 0.6)

Только через примитив иконки `<icon-primitive>` с `name=...` из реестра `<icon-registry>`. **НИКОГДА** не вставлять сырые юникод-кодпоинты (`\uXXXX`, `&#xf054;`) или классы иконочных шрифтов (`fa-*` и т.п.). Нужной иконки нет — **СТОП** (§2.1): предложить добавить с семантическим kebab-case именем и вариантом (light/solid/brand), дождаться решения.

### 2.5 Типографика (Профиль 0.3)

Для ВСЕХ размеров текста — utility-классы размера `<type-class>`. **НИКОГДА** не хардкодить `font-size` / `font-weight` / `line-height` в стилях.

Две независимые оси:
- **Семантический тег** (`h1`/`h2`/`h3`) отвечает за доступность и структуру документа.
- **Класс размера** `<type-class>` отвечает за визуальный размер.

Это РАЗНЫЕ оси: можно `h2` с классом размера body, и наоборот. Канон-роли размеров: `display / h1 / h2 / h3 / body / small / caption`. Inline-модификаторы: `strong` (вес), `mono` (моноширинный). T-shirt-варианты (`type-h1-large`) НЕ вводим.

### 2.6 Цвет текста (Профиль 0.3)

Inline-цвет текста — классы цвета `<text-color-class>` (`success / warning / danger / info / primary / muted`). **НИКОГДА** не хардкодить `color:`. Для контейнеров с заливкой — роль бейджа `<badge>` / callout `<callout>`, а не «класс цвета + кастомный фон». Числовые столбцы — класс `<tabular-class>`.

## 3. ПРАВИЛО КОМПОЗИЦИИ (нет CSS вне UI-слоя)

Файлы вне UI-слоя (то есть в `<views-dir>` и `<components-dir>/<domain>`) **НЕ МОГУТ** содержать:

- **блоки стилей** любого вида (scoped / module / plain — любой механизм стилей фреймворка);
- **инлайн-атрибуты стиля** (`style="..."`);
- **биндинги стиля** в любой форме (`:style="..."`, `style={{...}}`, в т.ч. для CSS-переменных).

Вся визуальная структура — композиция из примитивов UI-слоя. Не выражается через существующие примитивы — **СТОП** (§2.1) — предложить расширить/завести примитив — ждать решения.

**Динамические значения** (ширина прогресс-бара, цвет аватара по первой букве, координаты тултипа) живут ВНУТРИ примитивов UI-слоя, где биндинг стиля с CSS-переменной — правильный паттерн. Снаружи примитива динамику передаём как **ПРОП**, а не как инлайн-стиль.

Нейтральный псевдокод (динамику отдаём примитиву пропом, не стилем):

```
// СНАРУЖИ (страница/фиче-блок) — НЕТ стиля, только проп:
<ProgressBar value={percent} />
<Avatar letter={user.name[0]} color={user.color} />

// А так НЕЛЬЗЯ снаружи примитива:
//   React:  <div style={{ width: `${percent}%` }} />
//   Vue:    <div :style="{ width: percent + '%' }" />

// ВНУТРИ примитива ProgressBar (UI-слой) — здесь биндинг стиля законен:
//   React:  <div className="bar" style={{ ['--value']: percent + '%' }} />
//   Vue:    <div class="bar" :style="{ '--value': percent + '%' }" />
//   + в CSS примитива:  .bar::after { width: var(--value) }
```

**Parent-owns-gap.** Отступы между блоками задаёт **РОДИТЕЛЬ** — через `gap` на flex/grid-контейнере (или через раскладку-стек `<stack>` с пропом `gap`), а **НЕ** ребёнок через `margin`. Никаких `margin-top` / `margin-bottom` на корне компонента. Разрешено: `margin: 0` (reset) и отрицательные margin (sr-only, намеренный bleed).

**Исключения из запрета на CSS** (Профиль 0.8): сам UI-слой `<ui-dir>`, корень приложения / точка входа `<app-root>`, app-chrome синглтоны (`<app-sidebar>` / `<app-topbar>` / `<app-footer>` / `<app-bottom-nav>`). **Их ДЕТИ — НЕ исключение.**

**Внутри UI-слоя тоже нет хардкода:** цвета — только через токен `<color-token>`; spacing / radius / типографика / тень — только через токены (`<space-token>` / `<radius-token>` / `<text-token>` / `<shadow-token>`). Нет токена — **завести токен** (в `:root` + override темы, напр. `[data-theme="light"]`), а НЕ вписывать hex/px в компонент.

**DRY.** Один блок стиля повторился в двух местах — вынести в общий стиль/примитив, не копипастить.

## 4. КОНВЕНЦИИ API ПРИМИТИВОВ

| Ось | Правило | НИКОГДА |
|---|---|---|
| Размер (`size` на кнопке, аватаре, модалке…) | только `sm` / `md` / `lg`, дефолт `md`; примитив ОБЯЗАН работать без пропа; `md` = «стандарт для ЭТОГО примитива», не глобальные px | `xs` / `xl` / `2xl`, слова (`medium`), цифры |
| Spacing / gap (раскладка) | шкала шире: `xxs / xs / sm / md / lg / xl / 2xl`, дефолт `md`; `xxs` (≈4px) — hairline между тесно связанными строками | — |
| Ось `color` | семантика: `default / primary / success / warning / danger` | — |
| Ось `variant` | визуальная форма: кнопка `solid/light/link`; карточка `default/dashed`; иконка `light/solid/brand` | — |
| Не схлопывать оси | `color` и `variant` — ДВЕ независимые оси | один проп; третье имя `tone` / `kind` / `appearance` |
| Состояния | boolean-пропы причастиями/прилагательными: `disabled`, `loading`, `active`, `selected`, `expanded` | существительные |
| Дефолты пропсов | в механизме дефолтов фреймворка | `?? 'value'` в шаблоне/JSX |

Дефолты задаются механизмом дефолтов фреймворка (имя механизма — см. стек в Профиле 0.1), НЕ в разметке. Ниже показаны оба стека наравне — ядро ни к одному из них не привязано:

```
// Vue
withDefaults(defineProps<{ size?: 'sm' | 'md' | 'lg' }>(), { size: 'md' })

// React (значение по умолчанию параметра / деструктуризация с дефолтом)
function Button({ size = 'md' }: { size?: 'sm' | 'md' | 'lg' }) { /* ... */ }
```

Не вписывается в конвенции — **СТОП**, предложить, ждать. Не изобретать ad-hoc API.

## 5. HTTP (Профиль 0.4)

Любой запрос «SPA → backend» идёт через единую обёртку над fetch `<api-fetch>` (файл `<api-client>`).

Прямой `fetch()` из компонента — **антипаттерн**: без базового префикса `<api-base>` легко словить «404 на dev-server порту `<dev-port>`»; обёртка сама ставит префикс, `Authorization`, request-id (Профиль 0.4) и единый error-shape.

- JSON GET/POST → `<api-fetch>({ method, path, body })`.
- Multipart → `<api-fetch>({ method, path, bodyForm: fd })`.
- Скачивание бинарника → backend отдаёт JSON `{ url }`, фронт открывает presigned URL (он уже авторизован) — `window.open(url)`.

Кажется, что обёртка чего-то не умеет — сначала прочитать её исходник, потом расширить, и только в последнюю очередь городить параллельный путь.

Все HTTP-вызовы — в типизированном клиенте `<api-dir>/<domain>`, по доменам. **НЕ из компонента напрямую.**

## 6. КОНТРАКТ LIST-ЭНДПОИНТОВ И ТАБЛИЦ

### 6.1 Перед написанием страницы со списком — ответить на 3 вопроса

1. **Сколько строк на горизонте 6 месяцев?** Больше 100 — server-side фильтр ОБЯЗАТЕЛЕН.
2. **Что пользователь реально делает:** browse / search / autocomplete / админ-CRUD?
3. **Есть ли похожий по cardinality + UX список?** (похожий API ≠ похожие constraints).

### 6.2 Категории ресурсов → лимиты / пагинация / total

| Категория | Пример | Default limit | Max | Пагинация | Total |
|---|---|---|---|---|---|
| Browseable | orders, audit, products | 10 | 100 | cursor | через `has_more` |
| Reference search-driven | categories, suppliers | 10 | 100 | offset/page | да |
| Autocomplete | подсказки по вводу | 20 | 50 | нет | нет |
| Closed-tiny (≤20) | proxies, города | без лимита | — | нет (client-filter OK) | нет |

Единый envelope ответа — `{ data, meta }` (Профиль 0.7). На фронте — типизированный конверт списка `<list-response>` (`ListResponse<T>` / `ListMeta`), флаг «есть ещё» `<has-more>`, request-id `<request-id>` для трассировки.

### 6.3 Hard-stop анти-паттерны

| Анти-паттерн | Почему запрещён |
|---|---|
| `listXxxAll()` на клиенте, в цикле фетчащий все страницы, чтобы собрать ВЕСЬ список | собирает всё в память; не масштабируется |
| client-side фильтр по полному справочнику при cardinality > 100 | тянет весь справочник на клиент |
| реакция на 502/OOM подъёмом лимита | лимит — это **диагноз**, а не лечение |
| копипаст соседнего list-handler без проверки cardinality | похожий API ≠ похожие constraints |

### 6.4 4-state контракт list-view

Ровно **4 взаимоисключающих состояния** — `loading / error / empty / loaded`.

| Состояние | Где живёт |
|---|---|
| loading | ВНУТРИ таблицы-списка `<table>` (проп `loading`) |
| error | ВНУТРИ таблицы-списка `<table>` (проп `error` + повтор через механизм событий/колбэк фреймворка) |
| empty | отдельная ветка через empty-state `<empty>` — ТОЛЬКО когда fetch УСПЕШЕН и пусто |
| loaded | данные в таблице |

> **Внимание:** если эндпоинт отдаёт `.error` — caller **ОБЯЗАН** прокинуть его в проп `error` таблицы, иначе ошибка молча станет empty-state и баг спустится вниз. Повтор (retry) caller обрабатывает через механизм событий/колбэка своего фреймворка — конкретное имя события задаётся стеком (Профиль 0.1), не ядром.

### 6.5 Матрица error-состояний

| Где случилось | Что показать |
|---|---|
| inline в форме / под полем | callout `<callout>` (`color=danger`) |
| внутри списка | проп `error` на таблице-списке `<table>` + повтор через событие/колбэк фреймворка |
| page-level prerequisite-fetch упал | error-state `<error>` (полноэкранный) |
| транзиентный фидбек (после клика) | тост `<toast-fn>` |
| критическая ошибка с request-id для рапорта | `<alert-error-fn>` |

**Коммит = логическая единица работы, не каждое нажатие клавиши.**

- В длинных сессиях после каждой существенной посадки (новый модуль/экран/примитив, смена навигации) — **проактивно предложить чекпойнт**.

- Эвристика «trivial»: если в self-review нечего написать — trivial, батчим.
